  `metadata.task_states.unsqueeze(2).expand(-1, -1, steps_per, -1)`.

Packing into network input:

  `packed = net.pack_inputs(stim_seq, cue_seq, task_seq)`.
- src/model/network.py:169-190 (`pack_inputs`) —
  concatenates along the feature dim into [B, T, N+N+2].

  `task_state_seq = packed_input[:, :, 2*N:]` unpacked per-step.
- src/model/network.py:264 — passed into step():
  `self.step(stimulus_seq[:,t], cue_seq[:,t], task_state_seq[:,t], state)`

Consumption in V2 (ONLY consumer inside the network):
- src/model/network.py:123-125 — `self.v2(r_l4, state.r_l23, cue,
  task_state, state.h_v2)`.
- src/model/v2_context.py:94-99 — task_state is concatenated into the
  GRU input vector:

      if self.v2_input_mode == 'l23':
          v2_input = torch.cat([r_l23_prev, cue, task_state], dim=-1)
      elif self.v2_input_mode == 'l4':
          v2_input = torch.cat([r_l4, cue, task_state], dim=-1)
      elif self.v2_input_mode == 'l4_l23':
          v2_input = torch.cat([r_l4, r_l23_prev, cue, task_state], dim=-1)

  The GRU input_dim in all three modes ends in `+ 2` (v2_context.py:40/42/44).

Critical dead-ends:
- task_state does NOT reach the feedback E/I split in network.step()
  (lines 135-141 use only `feedback_scale` and the raw V2 head output).
- task_state does NOT reach the loss function.
  Verified: `grep -n task_state src/training/losses.py` → 0 matches.
  `loss_fn(...)` call in stage2_feedback.py:461-474 has no task_state
  keyword.

Consequences: task_state can only influence the network through whatever
V2 learns to encode in its hidden state. There is NO path from task_state
to a per-sample loss weight, to a feedback gain, or to the L2/3 E/I
balance. Any switch behaviour has to be implicit in V2's feedback_signal
output.

Changes needed for a task-state-conditioned regime switch:
- Make task_state informative during training (currently independent of
  stimulus distribution): either (a) correlate task_state with cue
  validity / ambiguity in sequences.py, or (b) interleave homogeneous
  task_state batches that differ in loss weighting.
- Route task_state into the E/I split (network.py:135-138) as a
  multiplicative gain, OR mix it into pi_pred's role (see §2).
- Pass a [B] task_relevance vector into the loss module so that
  orientation-specific losses (sensory, sharp, local_disc) scale
  per-sample. Currently loss signature in losses.py:473 has no such
  argument.

## 2. pi_pred plumbing — where does the V2 precision actually land?
----------------------------------------------------------------------

Source:
- src/model/v2_context.py:102 —
  `pi_pred = torch.clamp(F.softplus(self.head_pi(h_v2)), max=self.pi_max)`.
  Shape [B, 1]. Returned as the 2nd element of V2.forward().

Consumption in network.step():
- src/model/network.py:123 unpack: `mu_pred, pi_pred_raw, feedback_signal, h_v2`
- src/model/network.py:131 — `pi_pred_eff = pi_pred_raw * self.feedback_scale`
  (feedback_scale is the warmup ramp buffer, lines 40-41).
- src/model/network.py:144 —
  `deep_tmpl = q_pred * pi_pred_eff`
  This is the ONLY downstream use of pi_pred_eff in the forward pass.

Crucially:
- Line 135 — `scaled_fb = feedback_signal * self.feedback_scale`
  The E/I split is scaled by `feedback_scale` (warmup), NOT by pi_pred.
- Lines 136-137 —
      center_exc  = torch.relu(scaled_fb)
      som_drive_fb = torch.relu(-scaled_fb)
  No pi_pred multiplier anywhere in the split.
- Line 141 — `r_l23 = self.l23(r_l4, state.r_l23, center_exc, r_som, r_pv)`
  L2/3 receives center_exc (no pi gate).

Where does deep_tmpl go?
- It is stored in NetworkState.deep_template (network.py:154) so that the
  forward() loop can stack it into `deep_template_all`.
- It is exported to outputs at stage2_feedback.py:381 —
  `"deep_template": aux["deep_template_all"]`.
- It then enters ONLY `energy_cost()` in losses.py (referenced via
  `outputs["deep_template"].abs().mean()` — from my earlier audit of
  losses.py). No other loss consumes it. It provides a weak L1 pressure
  on pi_pred toward zero and nothing else.

pi_pred in losses:
- `grep -n pi_pred src/training/losses.py` → 0 matches.
- pi_pred does NOT appear in any loss term directly.

Verdict (flags advisor claim):
- Advisor claim: "pi_pred scales the placeholder deep_template rather
  than the actual E/I split" — **CONFIRMED CORRECT**.
  pi_pred only multiplies the dead deep_tmpl placeholder; it has no
  causal effect on the center_exc / SOM-drive branches. The only way
  pi_pred influences training at all is through the weak energy L1 on
  `deep_template_all` — effectively a pressure toward zero that never
  shapes the feedback waveform.

Changes needed:
- If we want pi_pred to actually gate feedback, line 135 needs to become
  something like
      scaled_fb = feedback_signal * self.feedback_scale * pi_pred_eff
  (or gate only the center_exc branch, or only the SOM branch — each
  has different biological interpretations).
- Delete deep_tmpl (lines 144, 154) once pi_pred is repurposed, OR keep
  it as a debug readout only.

## 3. Feedback E/I split — current behaviour vs what a switch needs
----------------------------------------------------------------------

Current code (network.py:131-141):

    pi_pred_eff  = pi_pred_raw * self.feedback_scale          # [B, 1]
    scaled_fb    = feedback_signal * self.feedback_scale      # [B, N]
    center_exc   = torch.relu(scaled_fb)                      # [B, N]
    som_drive_fb = torch.relu(-scaled_fb)                     # [B, N]
    r_som        = self.som(som_drive_fb, state.r_som)        # [B, N]
    ...
    r_l23        = self.l23(r_l4, state.r_l23, center_exc, r_som, r_pv)

Properties:
- Symmetric split by sign. `feedback_signal` is one learned linear head
  (v2_context.py:55: `self.head_feedback = nn.Linear(H, N)`).
- Both halves gated by the SAME scalar warmup buffer `feedback_scale`.
  No separate excitation gain vs inhibition gain — all the asymmetry V2
  can produce must come from the sign and magnitude of the raw head.
- No pi_pred gating.
- No task_state gating.
- `populations.py:252` confirms L2/3 uses `template_modulation`
  (= center_exc) as a purely additive excitatory input; L2/3 knows
  nothing about task_state.

To express sharpening AND dampening in one checkpoint via task_state:
- Minimum plumbing change: insert a [B, 1] multiplicative gain into the
  E and I branches derived from task_state (and optionally pi_pred).
  Two credible parameterisations:
    (a) Asymmetric branch gains:
          g_E = f_E(task_state, pi_pred)   # [B, 1]
          g_I = f_I(task_state, pi_pred)   # [B, 1]
          center_exc   = g_E * torch.relu(scaled_fb)
          som_drive_fb = g_I * torch.relu(-scaled_fb)
        With `g_E high / g_I low` → sharpening; `g_E low / g_I high` →
        dampening. Both are differentiable; f_E, f_I can be a tiny MLP.
    (b) Rotation on the (exc, inh) plane:
          alpha = f(task_state, pi_pred) ∈ [0, 1]
          center_exc   = alpha * torch.relu(scaled_fb)
          som_drive_fb = (1 - alpha) * (...same raw signal or mirror)
- Either lives entirely inside network.step() at lines 135-141 and
  requires no changes to V2, losses, or stimulus generation.
- Must be combined with §4/§5 changes so the task_state bit is
  correlated with something learning can optimise.

----------------------------------------------------------------------

Loss signature (losses.py:473):

    def forward(self, outputs, true_theta_windows, true_next_theta_windows,
                r_l23_windows, q_pred_windows, state_logits_windows=None,
                true_states_windows=None, p_cw_windows=None, model=None,
                is_expected=None, predicted_theta=None, use_e_total=True,
                fb_scale=1.0, r_l4_windows=None, mismatch_labels=None,

There is NO `task_state` argument. All reductions are global:
- sensory_readout_loss → F.cross_entropy(..., reduction="mean")
- homeostasis_penalty → `.mean()`

Minimum-diff path for per-sample task-conditioned losses:
1. Add `task_state_windows=None` to loss_fn.forward() signature

2. Inside each orientation-specific term, compute a per-sample loss

## 5. Stimulus conditioning — where task_state is not (yet) informative
----------------------------------------------------------------------

HMMSequenceGenerator (sequences.py constructor, defaults):
- contrast_range             (inherited from TrainingConfig)
- ambiguous_fraction = 0.15
- ambiguous_offset = 15.0
- reliability_drop_prob = 0.05
- reliability_drop_value = 0.50
- cue_dim = 2
- n_states = 3
- cue_valid_fraction = 0.0

task_state bit generation (sequences.py:379-383): independent
Bernoulli(0.5), NOT conditioned on any of the above. Consequences:

- Training currently marginalises over task_state → V2 has no incentive
  to route information differently across the two task_state modes.
- Even if we wire task_state into the E/I split or into the loss, V2
  only has one functional niche to learn because both task_state values
  see the same stimulus distribution AND the same loss.

For a regime switch, task_state must co-vary with either (or both):
  (i) Loss weights (§4). Simplest: relevant ↔ lambda_sensory HIGH,
      irrelevant ↔ lambda_sensory LOW + lambda_energy HIGH. This maps
      directly to the Kok↔Richter distinction (predicted feature is
      task-critical vs orthogonal).
  (ii) Stimulus reliability. E.g., relevant samples come from the sharp
      end of ambiguous_fraction / cue_valid_fraction, irrelevant from
      the noisy end. This is biologically motivated (tasks change
      uncertainty structure) but requires mirroring the split inside
      sequences.py.

Minimum-diff path (loss-side only): zero changes to sequences.py if we
do the cue-free version. task_state stays as a random [B] bit; it still
enters V2's GRU (existing path) AND it routes loss weights per-sample.
The crossed design gets run at analysis time via TaskRelevanceParadigm
(§7).

## Minimum-diff path to a task-state-conditioned regime switch
----------------------------------------------------------------------

Phase 0 (sanity, no code): verify the current checkpoint. Run
`TaskRelevanceParadigm` on the e1 configuration. Expected result:
essentially no difference between relevant and irrelevant cells,
because task_state currently enters ONLY the V2 GRU and V2 has no
training incentive to use it. If this is confirmed, it falsifies the
"just train longer and it'll learn to switch" framing and motivates
Phase 1.

Phase 1 (loss-side only, simplest possible diff): make task_state
control per-sample loss weights. No architectural change.
  Touches:
    src/training/losses.py       — add task_state_windows arg to
                                    Stage2Loss.forward(); inside sensory
                                    / sharp / local_disc terms, compute
                                    per-sample losses and weight by a
                                    function of task_state.
    src/training/stage2_feedback.py:461-474 — pass task_seq window into
                                    loss_fn.
    src/training/trainer.py or equivalent — add an `extract_readout_data`
                                    step for task_seq (grab one row per
                                    presentation since it's sequence-
                                    constant; lines 186-187 make this
                                    trivial).
  Zero changes to: network.py, v2_context.py, populations.py,
  sequences.py, analyze_representation.py (the existing M8/M13 time
  windows are reusable).
  Hypothesis this tests: "V2's GRU input is sufficient to encode a
  task_state-conditioned feedback waveform if the training loss
  rewards different waveforms in the two modes." I rate this credible
  but NOT certain, because the GRU would have to learn to change
  `head_feedback` outputs as a function of a 2-dim input that enters
  additively; one linear head may not be expressive enough. Validator
  should check feedback_signal correlations with task_state post-train.

Phase 2 (only if Phase 1 is insufficient): add a multiplicative E/I
gain driven by (task_state, pi_pred).
  Touches:
    src/model/network.py:131-141 — replace lines 135-141 with the
                                    gated version (either (a)
                                    asymmetric branch gains or (b)
                                    rotation on the (exc, inh) plane
                                    from §3).
    src/model/network.py:__init__  — register the tiny gating MLP
                                    (1-2 params per branch).
    tests/test_network.py          — add shape + symmetry tests.
  This is the "pi_pred actually gates the E/I split, not just the
  placeholder deep_tmpl" change the advisor correctly identified.
  Pairs naturally with deleting the deep_tmpl placeholder (§2), though
  deep_tmpl can be left as a debug readout for continuity.

Phase 3 (only if Phases 1+2 do not cross the biological benchmark):
stimulus correlations. Correlate task_state with cue validity /
ambiguity fraction inside HMMSequenceGenerator so that V2 sees different
input statistics in the two modes. Minimally invasive version: leave
generation random but gate cue_seq or ambiguous_fraction by the sampled
task_state bit.

DO NOT recommend:
- Touching populations.py. None of Phases 1-3 require it.
- Re-introducing VIP to implement the switch until Phases 1 and 2 are
  proven insufficient — it is a bigger architectural change and the
  simpler gain-only mechanism may already be enough.
- Adding new loss terms before reusing the existing (lambda_sharp,
  lambda_local_disc) disabled-by-default machinery. They were built for
  exactly this kind of experiment.

## Risks / unknowns for the Lead
----------------------------------------------------------------------

1. Even with Phase 1 + Phase 2, the task_state bit is sequence-level
   (constant within a sequence, sequences.py:379-383). If you want
   *within-sequence* switching (e.g., task changes mid-trial), the
   generator has to be rewritten to produce a per-presentation
   task_state. This is a sequences.py + trainer.py change, not a
   network change.

2. The current `feedback_scale` warmup mechanism means that very early
   in training, E and I are both strongly damped. Any Phase 2 gain MLP
   will see near-zero gradient until feedback_scale reaches 1.0 at
   ~burnin_steps=5000 (config.py:80). Schedule the gain MLP LR and
   init accordingly.

3. `TaskRelevanceParadigm` uses a `deep_template` field inside
   ConditionData (paradigm_base.py:57) that is always zero in the
   current forward path; removing deep_tmpl in Phase 2 will break that
   dataclass. Minor — just make it optional.

4. Validator should add a regression test that `grep task_state
   src/training/losses.py` returns non-zero after Phase 1, and that
   `feedback_signal` per-sample mean differs between task_state=[1,0]
   and task_state=[0,1] after training in Phase 1. These are cheap
   guard rails that directly falsify the "nothing changed" failure
   mode.

5. Biological benchmark targets (Richter 2018 dampening ~M7=-0.02,
   M10=0.85; Kok 2012 sharpening ~M7=+0.10, fwhm narrowing ~12°) fall
   BETWEEN already-tested sweep rows (a1 dampening vs e1 sharpening).
   A single checkpoint has to reproduce BOTH cells in its crossed
   table — i.e., the per-sample loss split must be large enough to
   move e1-style M7 into a1-style M7 for the irrelevant half. Neither
   the sweep nor any existing run has demonstrated this kind of split
   is achievable from one set of weights. This is the key scientific
   risk of the whole plan, independent of plumbing.
